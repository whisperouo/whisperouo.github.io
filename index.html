技术难点

hive没办法创建分区了。重启了多次，都没有效果。后来发现namenode也一直出现故障。

解决方案

最终，锁定一个参数配置dfs.namenode.handler.count。如果该值设的太小，明显的状况就是DataNode在连接NameNode的时候总是超时或者连接被拒绝,后来本地的维护人员将这个值调到了4000,也就是说，配置了一个线程池，可以容纳4000个线程，这样在消息通信时，这里容纳的信息量就是非常庞大的，最终造成了OOM。因此这个值可以放大，但是不能加的过大。最终,按照dfs.namenode.handler.count=20 * log2(Cluster Size)这个公式配了50个线程则解决了这个问题

MR造成系统宕机

控制Yarn同时运行的任务数，和每个任务申请的最大内存

写入文件过量造成NameNode宕机
解决方案：调高Kafka的存储大小，控制从Kafka到HDFS的写入速度。高峰期的时候用Kafka进行缓存，高峰期过去数据同步会自动跟上。

用户session访问时长的计算
  1. 基于session粒度进行数据聚合
    • 聚合原因
      因为用户的筛选条件比较多，如日期，性别，搜索关键词等数据都是不同维度的，不统一维度的话，会导致全表搜索
    • 聚合过程
      1. 基于session聚合
        1. 利用mapPartitionsToPair算法获得sessionId 与action的tuple RDD
        2. 利用groupByKey 算子，聚合同一seesion的信息如点击、购买、收藏、搜索、访问时长用 | 分割统计出来拼接在一个字符串中
        3. 利用mapToPair 算子将其构造出userId为key，value为拼接的session信息的tuple，用mapToPair的原因是下一步中需要聚合user信息，因此用userId作为key
      2. 聚合user信息
        • 为了将用户搜索信息拼接，需要用上一步中得出的userId2SessionInfo join user 信息，这一步可能会出现数据倾斜，适合用map join 替换掉 reduce join，也就是将user信息作为广播变量广播出去，然后在利用userId2SessionInfo 在调用mapToPair算子的时候取出广播变量的值进行聚合user信息。
    • 聚合结果
      • 用户需要的搜索条件都被拼接在一起
  2. 自定义accumulator实现session访问时长统计
    • 利用filter算子先进行用户筛选条件的过滤
    • 自定义原因
      • 为了进行统计session，累加不同节点上同session时长的信息，就必须有共享变量，也就是使用自定义accumulator
    • 为什么不用传统方式
      • accmulator太多，不便于维护
        1. 可能累加错
        2. 项目逻辑变更时，维护成本太高
    • 我的方式
      • 自定义一个accmulator，而不是用传统的几十个
    • 注意
      • 在将accumulator统计的session时长写入mysql中，必须有1个action操作在前，不然任务不会被触发

  • 实现目的

    1. 首先自定义了车型排序key类实现Orderd接口，重写compare 、less、greater等方法

    4. 将上一步中形成的tuple RDD 利用mapToPair算子将点击、收藏、搜索次数提取出来映射成自定义的排序key，value为统计的信息

一、

自定义原因

Accumulator是运行再Driver端上了，内部维护了一个叫value的private变量，用来对作业执行过程中一些事件的统计。但是默认的只能对某一事件每次进行+1操作，也不能进行复杂的运算
如果采用默认Accumulator的做法是可以对不同的session时长都定义一个Accumulator来计算统计对应个数，但是这样会造成项目中大量的Accumulator存在不便维护，因此需要自定义Accumulator。

实现流程

实现AccumulatorParam接口，重写了Zero方法，初始的值是字符串，格式是key =value的形式，并且每个值用 | 分割，重写了addInplace方法和addAccumulator方法，两个方法内部调用逻辑是一样的，获取到传入的key，根据传入的key利用String split方法将对应的旧值拿出来转换成int类型之后+1然后拼接回去形成新的值给回到Accumulator，相当于改写成功了。

二、

自定义原因

sortByKey算子，默认情况下，它支持根据int、long等类型来进行排序，但是那样的话，key就只能放一个字段了所以需要自定义key，作为sortByKey算子的key，自定义key中，封装n个字段，并在key中，自己在指定接口方法中，实现自己的根据多字段的排序算法然后再使用sortByKey算子进行排序，那么就可以按照我们自己的key，使用多个字段进行排序

在Spark实时项目中，用户访问 session 通常指

用户在网站或应用中的一系列操作，这些操作在一段时间内被认为是一个连续的访问会话。会话一般是从用户进入网站或应用开始，到用户离开为止的一段时间范围。

用户访问 session 可以包括多个页面浏览、点击、搜索等行为。会话的定义通常基于一些时间间隔或用户活动的规则，比如一段时间内没有用户操作就认为会话结束。这个时间间隔可以根据具体项目的需求来定义。
在实时项目中，使用Spark Streaming等流处理框架来处理用户访问 session 数据是很常见的。通过实时处理，可以及时了解用户的行为，进行个性化推荐、实时分析等。通常，需要使用一些技术手段来识别和跟踪用户的 session，比如使用用户的ID或在用户访问时生成的会话标识符。
总的来说，用户访问 session 在实时项目中是一个时间上连续的用户操作序列，通过对这些操作进行处理和分析，可以从中获取有关用户行为的有用信息。

hdfs流程

分布式文件系统（Hadoop Distributed File System）

  • 读流程
  1. client查询元数据
    • 访问namenode，获得文件数据块位置列表，返回到输入流对象
  2. 就近挑一台datanode服务器，请求建立建立输入流
  3. datanode向输入流中写数据，以packet为单位校验
  4. 关闭输入流
  • 写流程
  1. client向namenode发出写文件请求
  2. 检查是否已存在文件、检查权限，若通过检查，先将操作写入editlog，并返回输出流对象
    • editlog记录的是最新的HDFS客户端执行的所有写操作，如果后续实际写操作失败了，也会有记录，不用担心后续client读不到相应数据块
    • 第5步中datanode收到块后会返回确认信息，若发送端收不到确认信息，会一直重试，直到成功
  3. client端按照128MB切分文件
  4. client将namenode返回的分配可写的datanode列表和data数据一同发送给最近的第一个datanode节点，此后client端和namenode分配的多个datanode组成pipeline管道，client端向输出流对象中写数据
    • client每向第一个datanode写入一个packet，这个packet便会直接在pipeline中传给第二个第三个…datanode（并不是写完一个block块或一整个文件后才发）
  5. 每个datanode写完一个block块后，返回确认信息
  6. 写完数据关闭输出流
  7. 发送完成信号给namenode
    • 发送信号时机取决于集群是
      • 强一致性（一般是）：所有datanode写完，向namenode汇报
      • 最终一致性：任意一个datanode写完，单独向namenode汇报
  • block
    • 最大
    • 文件上传前分的块
    • 一般128MB（可以改，不推荐）
      • 块太大：Map任务数少，执行速度变慢
      • 块太小：寻址时间占比过高
  • packet
    • 第二大
    • client向datanode、datanode的pipeline之间 传数据 的基本单位
    • 默认64KB
  • chunk
    • 最小
    • client向datanode、datanode的pipeline之间 数据校验 的基本单位
    • 默认512Byte
    • 实际516Byte（512+4Byte校验位）

yarn流程

可回答：1）YARN的任务提交流程；2）YARN的通信流程或工作流程；3）YARN执行一个application的流 程；4）Hadoop任务的YARN调度过程；5）YARN上提交资源（程序）的流程；6）YARN的资源调度流程；7）YARN的任务提交流程

1）MapReduce程序提交到客户端所在的节点。
2）YarnRunner向ResourceManager申请一个Application。
3）RM 将该应用程序的资源路径以及application_id返回给YarnRunner。
4）该程序将运行所需资源提交到 HDFS 上。
5）程序资源提交完毕后，申请运行 mrAppMaster。
6）RM 将用户的请求初始化成一个 Task。
7）其中一个NodeManager 领取到 Task 任务。
8）该 NodeManager 创建容器 Container，并产生 MRAppmaster。
9）Container 从HDFS 上拷贝资源到本地。
10）MRAppmaster 向RM 申请运行 MapTask 资源。
11）RM 将运行 MapTask 任务分配给另外两个 NodeManager，另两个 NodeManager 分别领取任务并创建容器。
12）MR 向两个接收到任务的NodeManager 发送程序启动脚本，这两个 NodeManager分别启动MapTask，MapTask 对数据分区排序。
13）MrAppMaster 等待所有 MapTask 运行完毕后，向 RM 申请容器，运行 ReduceTask。
14）ReduceTask 向 MapTask 获取相应分区的数据。
15）程序运行完毕后，MR 会向 RM 申请注销自己。

    • 每个Task执行结果就是生成了目标RDD的一个partition
    • 在parallelize方法的逗号右边设置的数字（分区数）即task个数（其等于设置的RDD并行度）

数据倾斜

本质

某个或某些key对应的数据远远高于其他key

举例

出现原因

基本是shuffle导致

可能触发数据倾斜的算子

可能触发的算子（distinct、groupByKey、reduceByKey、 aggregateByKey、join、cogroup、repartition）

如何定位

  1. 找找程序中使用的可能产生shuffle的算子
  2. 看log
    • 一般会报哪一行代码导致OOM异常
    • 看看执行到第几个stage
      • 哪一个stage的task特别慢，然后定位

解决办法（两个）

  1. 使用Hive ETL预处理数据
  2. 过滤导致倾斜的key

一、使Hive ETL预处理数据

方案适用场景：导致数据倾斜的是Hive表。如果该Hive表中的数据本身很不均匀（比如某个key对应了100万数据，其他key才对应了10条数据），而且业务场景需要频繁使用Spark对Hive表执行某个分析操作，那么比较适合使用这种技术方案。

方案实现思路：此时可以评估一下，是否可以通过Hive来进行数据预处理（即通过Hive ETL预先对数据按照key进行聚合，或者是预先和其他表进行join），然后在Spark作业中针对的数据源就不是原来的Hive表了，而是预处理后的Hive表。此时由于数据已经预先进行过聚合或join操作了，那么在Spark作业中也就不需要使用原先的shuffle类算子执行这类操作了。

方案实现原理：这种方案从根源上解决了数据倾斜，因为彻底避免了在Spark中执行shuffle类算子，那么肯定就不会有数据倾斜的问题了。但是这里也要提醒一下大家，这种方式属于治标不治本。因为毕竟数据本身就存在分布不均匀的问题，所以Hive ETL中进行group by或者join等shuffle操作时，还是会出现数据倾斜，导致Hive ETL的速度很慢。我们只是把数据倾斜的发生提前到了Hive ETL中，避免Spark程序发生数据倾斜而已。

方案优点：实现起来简单便捷，效果还非常好，完全规避掉了数据倾斜，Spark作业的性能会大幅度提升。

方案缺点：治标不治本，Hive ETL中还是会发生数据倾斜。

方案实践经验：在一些Java系统与Spark结合使用的项目中，会出现Java代码频繁调用Spark作业的场景，而且对Spark作业的执行性能要求很高，就比较适合使用这种方案。将数据倾斜提前到上游的Hive ETL，每天仅执行一次，只有那一次是比较慢的，而之后每次Java调用Spark作业时，执行速度都会很快，能够提供更好的用户体验。

项目实践经验：在美团·点评的交互式用户行为分析系统中使用了这种方案，该系统主要是允许用户通过Java Web系统提交数据分析统计任务，后端通过Java提交Spark作业进行数据分析统计。要求Spark作业速度必须要快，尽量在10分钟以内，否则速度太慢，用户体验会很差。所以我们将有些Spark作业的shuffle操作提前到了Hive ETL中，从而让Spark直接使用预处理的Hive中间表，尽可能地减少Spark的shuffle操作，大幅度提升了性能，将部分作业的性能提升了6倍以上。

HiveSQL

——AST（抽象语法树）

——QB（查询块）

  1. Hive根据Antlr定义的词法、语法规则完成词法、语法分析将HQL解析为AST Tree
  2. 遍历AST Tree，抽象出查询的基本组成单元Query Block
  3. 遍历Query Block解析为操作树Operator Tree(即，逻辑执行计划)

  5. 遍历Operator Tree，将操作树翻译为对应的MapReduce任务
  6. 物理优化器进行MapReduce任务变换，生成最终的执行计划

数仓理论

分层的意义 四个

    1. 复杂问题简单化
    • 复杂问题分解为多层，每层只处理简单任务，方便定位问题
    1. 减少重复开发
    • 通过中间层数据，极大的减少重复计算，增加一次计算结果的复用性
    1. 隔离原始数据
    • 使真实数据与统计数据解耦开
    1. 提高数仓可维护性
    • 某层问题只在该层解决，无需更改其它层代码或逻辑

分层结构

  • ADS层（Application Data Store）
    • 为各种统计报表提供数据
  • DWT层（Data WareHouse Topic）
    • 累计汇总
    • 一行信息代表一个主题对象的累计行为（如一个用户从注册那天至今一共下了多少次单）
  • DWS层（Data WareHouse Summary）
    • 按天汇总
    • 一行信息代表一个主题对象一天的汇总行为（如一个用户一天的下单次数）
  • DWD层（Data WareHouse Detail）
    • 清洗ODS层数据（去除空值、脏数据、超过极限范围的数据）、脱敏等
    • 一行信息代表一次业务行为（如一次下单）
  • ODS层（Operation Data Store）
    • 原始数据

常见模型介绍 两个

星型模型

  • 最常见
  • 结构简单，易于理解和实现
  • 组成
    • 事实表（Fact Table）
      • 模型中心
      • 存储事务性数据和度量指标（例如销售额、交易数量）
      • 每条记录通常有一个或多个指向维度表的外键
    • 维度表
      • 围绕事实表排列
      • 存储特定的属性信息（为事实表中的度量提供上下文）
      • 常见的包括时间、地点、客户、产品等
      • 含有多个列，每列是该维度的一个属性
  • 优点
    • 结构简单，查询快速。
    • 用户只需查询少数几个表就能获取所需信息
  • 适用于
    • 大规模、复杂的操作，尤其是用户需要从多个维度分析数据时

雪花模型

  • 结构更复杂，存储效率更高，查询效率可能更低
    • 星型模型的基础上增加了更多的层次和细节
  • 与星型模型的区别
      1. 更细更规范化的维度表（如地址信息被细分为“国家”“省份”）
      1. 更复杂的结构：查询涉及的表可能更多，性能有所下降
  • 优点
    • 数据的维护和存储效率更高，因为减少了数据冗余
    • 其规范化程度更高，有助于节约存储空间并保持数据的一致性
  • 适用于
    • 维度属性具有明显层次结构的场景

外键

数据库外键（Foreign Key)是关系型数据库的一个重要概念，它用于建立表与表之间的关系，以保证数据的完整性和一致性。

外键是指一个表中的一个或多个字段，它们的值必须在另一个表中的某个字段中存在。这个被参照的表中的字段通常是主键（Primary Key），这样就可以通过外键将两个表关联起来。

举个例子，假设有两个表，一个是订单表，一个是客户表。订单表中有一个字段是客户ID，这个字段就可以作为外键，参照客户表中的客户ID字段。这样，当我们在订单表中插入一条数据时，就必须保证这个客户ID在客户表中已经存在，否则就会出现数据不一致的问题。

外键的作用是保证数据的完整性和一致性。通过外键，我们可以将多个表关联起来，从而避免数据重复和冗余。同时，外键也可以限制数据的删除和修改，以避免对关联表中的数据造成影响。

需要注意的是，外键的使用需要谨慎，因为它会对数据库的性能产生影响。

  • 列不允许null

分治策略

适用于大数据

如何平均分配？

若数据分布不均衡，则先粗略分布到k个桶中，再把桶内元素太多的继续划分几个桶，直至所有桶数据大致均衡

基数排序

思想：

与计数排序一致，统计个数实现排序。在此基础上，利用数字位之间的递进关系，对每一位进行排序

适用于数值范围大的情况，但数据必须可以表示为固定位数的格式

时空复杂度

时间复杂度

  • 本质

    • 统计的不是算法运行时间，而是算法运行时间随着数据量变大时的增长趋势

如时间复杂度为“常数阶”，指的是算法运行时间不随着n增大而增大，始终为一个固定值

若为“线性阶”（n），则为线性增长


对数阶增长缓慢，仅次于常数阶

阶乘阶

通常使用递归实现

n大于等于4时，阶乘比指数增长更快

通常用最差时间复杂度作为评判标准

  • 计算方法

    • 主要看循环，并列是相加，嵌套是相乘

空间复杂度

空间复杂度：衡量算法占用内存空间随着数据量变大时的增长趋势

通常只关注最差空间复杂度

数学基础知识

质数（素数）

  • 大于1的自然数中，除了1和它本身外没有其他因数的自然数

自然数

  • 从0开始，one by one

质因数（素因数 or 质因子）

  • 能整除给定正整数的质数

互质

  • 两个没有其他共同质因数的正整数，1与其他正整数（包括1本身）都互质

丑数

  • 不能被2、3、5以外其他质数整除的数（1是丑数），即分解后含有2、3、5（任意一个）

第一步

下载并安装nodejs：https://nodejs.org/en

下载完成后会得到两个东西

node.js是本体
npm是包管理器，一会用于下载hexo框架

node -v
npm -v

终端输入指令：
npm install -g hexo-cli

-g应该是全局安装的意思

输入
hexo -v
看看是否正常输出

找个地方建空文件夹

打开终端

hexo初始化：hexo init
